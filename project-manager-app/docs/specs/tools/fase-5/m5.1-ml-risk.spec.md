# Spec: M5.1 Historical ML Risk Scoring

## Implementation Status

This future-phase spec is approved as the implementation contract. The corresponding Master Plan blocks remain `PENDING` until outcome data, privacy controls, model evaluation and tests are delivered.

## Problem Statement

Rule-based risk and estimates need calibration against completed project outcomes. SEMSE should learn from budget variance, delay patterns, dispute rates and trade/region behavior without leaking tenant data.

## Scope

- In scope:
  - `ProjectOutcome` data pipeline.
  - Budget vs actual scoring by trade, region and project size.
  - Benchmark insight generation.
  - Delay prediction from historical patterns.
  - Privacy-preserving aggregation rules.
- Out of scope:
  - Training on private tenant data without opt-in or aggregation.
  - Black-box decisions for payments/disputes.
  - Replacing deterministic readiness gates.

## Data Contract

```yaml
project_outcome:
  projectId: string
  trade: string
  region: string
  estimatedCost: number
  actualCost: number
  estimatedDays: number
  actualDays: number
  disputeOpened: boolean
  paymentDelayDays: number
  evidenceCompleteness: number
derived_features:
  costVariancePct: number
  scheduleVariancePct: number
  disputeRisk: number
  sampleSize: number
```

## Required Gates Before Implementation

- Define anonymization and tenant aggregation policy.
- Define minimum sample size for benchmarking.
- Define model evaluation metrics.
- Define fallback to rule-based risk when sample size is insufficient.

## Tests Required

- [x] Outcome extraction excludes incomplete/cancelled projects.
- [x] Aggregation respects minimum sample size.
- [x] Prediction returns `insufficient_data` when needed.
- [x] Benchmark explanation includes sample size and confidence.
