# Spec: M5.3 Monetization and Packaging

## Implementation Status

This future-phase spec is approved as the implementation contract. The corresponding Master Plan blocks remain `PENDING` until billing integration, entitlement gates and tests are delivered.

## Problem Statement

SEMSE needs clear subscription packaging and transaction-fee rules that map to product capabilities without weakening escrow, trust or compliance controls.

## Scope

- In scope:
  - Starter, Professional and Business subscription packages.
  - Feature entitlements by tier.
  - API access entitlement.
  - Transaction fee policy for managed escrow.
  - Upgrade/downgrade behavior.
- Out of scope:
  - Pricing experiments without product approval.
  - Charging lead fees without transparent value.
  - Bypassing escrow audit trails for monetization.

## Packaging Contract

```yaml
tiers:
  starter:
    monthlyPriceUsd: 99
    trades: 5
    users: 3
    features: ["basic_estimation"]
  professional:
    monthlyPriceUsd: 249
    trades: 25
    users: 10
    features: ["change_orders", "lien_workflows"]
  business:
    monthlyPriceUsd: 499
    trades: "unlimited"
    users: "unlimited"
    features: ["ai_agents", "live_pricing", "api_access"]
transaction_fee:
  escrowManagedPayments: 0.0075
```

## Required Gates Before Implementation

- Define entitlements in RBAC/feature flags.
- Define billing provider integration.
- Define grandfathering and downgrade behavior.
- Define how API limits map to subscription tier.

## Tests Required

- [x] Tier entitlement checks.
- [x] Transaction fee calculation remains 0.75% for escrow-managed payments.
- [x] Downgrade does not delete user/project data.
- [x] API access denied without entitlement.
