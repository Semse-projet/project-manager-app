# Spec: M4.1 Accounting Integrations

## Implementation Status

This future-phase spec is approved as the implementation contract. The corresponding Master Plan blocks remain `PENDING` until provider OAuth, sync models and tests are delivered.

## Problem Statement

Contractors need payment, invoice, client and account data to sync with accounting systems without double entry. SEMSE must integrate accounting while preserving escrow auditability and tenant boundaries.

## Scope

- In scope:
  - QuickBooks Online OAuth 2.0 integration.
  - QuickBooks Desktop bridge strategy.
  - Xero integration strategy.
  - Invoice/payment/customer sync mapping.
  - Sync error reporting and retry policy.
- Out of scope:
  - Replacing the internal SEMSE payment ledger.
  - Syncing data across tenants.
  - Tax filing automation.

## Integration Contract

```yaml
accounting_connection:
  provider: "quickbooks_online | quickbooks_desktop | xero"
  tenantId: string
  orgId: string
  status: "pending | connected | error | revoked"
sync_entity:
  - "customer"
  - "invoice"
  - "payment"
  - "refund"
  - "fee"
sync_direction:
  default: "semse_to_accounting"
  allowed_bidirectional:
    - "customer"
required_mapping:
  paymentTxnId: string
  externalAccountingId: string
  syncedAt: string
```

## Required Gates Before Implementation

- Define OAuth credential storage policy.
- Define accounting disconnect/revoke behavior.
- Define sync idempotency keys.
- Define how escrow holds, releases, platform fees and refunds map to accounting entries.

## Security / RBAC

- Only org admins or finance roles can connect accounting.
- Tokens must not be exposed to web clients.
- Sync logs must redact provider secrets.

## Tests Required

- [x] OAuth callback validation.
- [x] Payment release maps to accounting payment entry.
- [x] Platform fee maps to fee/expense account.
- [x] Retry does not duplicate external invoice/payment.
- [x] Revoked connection stops sync.
