# Spec: M4.2 Geolocation and Permits

## Implementation Status

This future-phase spec is approved as the implementation contract. The corresponding Master Plan blocks remain `PENDING` until provider integrations, privacy policy and tests are delivered.

## Problem Statement

Project location affects crew routing, evidence validity, permit requirements and inspection risk. SEMSE needs a formal contract for geocoding, permit tracking and optional aerial context.

## Scope

- In scope:
  - Google Maps project geocoding.
  - Field crew directions/location context.
  - OpenGov permit/inspection tracking strategy.
  - Google Aerial View context for estimation.
  - Permit risk surfaced to ProTools and BuildOps.
- Out of scope:
  - Automatic permit filing.
  - Legal code interpretation without human review.
  - Storing continuous worker location without consent.

## Data Contract

```yaml
project_location:
  address: string
  city: string
  state: string
  postalCode: string
  lat: number
  lng: number
permit_signal:
  projectId: string
  jurisdiction: string
  permitType: string
  status: "unknown | likely_required | submitted | approved | rejected | expired"
  source: "opengov | manual | rules"
inspection_signal:
  required: boolean
  deadline: string | null
  source: string
```

## Required Gates Before Implementation

- Define location consent and retention policy.
- Define provider fallback when Google/OpenGov unavailable.
- Define what permit status can block milestone/payment readiness.

## Tests Required

- [x] Address geocoding maps to project location.
- [x] Permit required signal propagates to risk/evidence requirements.
- [x] Missing provider data results in `unknown`, not false approval.
- [x] Location data remains tenant-scoped.
