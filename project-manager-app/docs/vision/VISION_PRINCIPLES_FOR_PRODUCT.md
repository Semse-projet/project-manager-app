# Vision Principles For Product

## Objetivo

Traducir la vision estrategica a principios concretos de producto.

Estos principios sirven para decidir:

- que construir;
- que no construir;
- como priorizar UX;
- como evitar features bonitas pero vacias.

## Principio 1. El producto debe reducir incertidumbre

Cada feature importante debe ayudar a responder mejor:

- que trabajo se va a hacer;
- quien lo hace;
- en que estado esta;
- que evidencia existe;
- que dinero esta retenido o liberado;
- que pasa si algo falla.

Si una feature no reduce incertidumbre, su valor es sospechoso.

## Principio 2. La interfaz debe reflejar la verdad operativa

La UI no debe inventar estados optimistas ni simplificaciones engañosas.

Debe reflejar:

- permisos reales;
- ownership real;
- estados reales del backend;
- bloqueos reales del flujo;
- errores reales como `403`, `404`, `409`.

## Principio 3. El dinero no puede ser una caja negra

Todo lo financiero debe ser comprensible para el usuario.

El producto debe mostrar con claridad:

- fondos fondeados;
- fondos retenidos;
- fondos liberados;
- fondos en disputa;
- reglas de bloqueo o liberacion.

## Principio 4. La evidencia no es adjunto, es infraestructura

El producto debe tratar la evidencia como parte central del flujo.

Eso implica:

- pedirla en el momento correcto;
- estructurarla;
- vincularla a hitos;
- facilitar revision;
- usarla en disputas y trust.

## Principio 5. Cada rol ve una verdad distinta del mismo sistema

El producto no debe intentar una sola interfaz mental para todos.

Debe asumir:

- cliente quiere control, claridad y seguridad;
- profesional quiere fluidez, pago justo y menos friccion;
- ops quiere trazabilidad, alertas y capacidad de intervenir;
- governance futura quiere explicabilidad y reglas claras.

## Principio 6. Primero confianza, luego velocidad

Si hay conflicto entre:

- hacer el flujo mas rapido;
- o hacerlo mas confiable y explicable;

en las partes sensibles debe ganar la confianza.

## Principio 7. El MVP debe sentirse pequeno pero serio

No debe parecer:

- una demo;
- una app a medias;
- un mock visual.

Debe parecer un sistema pequeno, enfocado y confiable.

## Principio 8. IA solo donde mejora trabajo real

El producto no debe meter IA por ornamento.

La IA debe entrar para:

- redactar mejor;
- detectar faltantes;
- resumir;
- alertar;
- acelerar soporte;
- mejorar supervision.

## Principio 9. Los errores deben ayudar a decidir

Cuando algo no se permite, el producto debe dejar claro:

- que paso;
- por que;
- que falta para avanzar;
- quien puede resolverlo.

## Principio 10. El largo plazo debe sentirse posible desde el diseno inicial

Aunque el MVP sea pequeno, el producto debe sugerir:

- seriedad;
- trazabilidad;
- institucionalidad;
- escalabilidad de reglas;
- capacidad de evolucion.

No por agregar complejidad visible, sino por consistencia.
