# Architecture Bridge Principles

## Executive summary

SEMSE should not abandon the previous path.
The previous path already created:
- a strong ecosystem vision;
- strategic direction;
- multiple working product pieces;
- a practical understanding of where the platform is going.

The new path adds:
- cleaner role structure;
- cleaner toolkit structure;
- clearer module ownership;
- clearer permission boundaries;
- a more formal construction operating system model.

## Bridge principle

The new path is not a replacement.
It is an organizing layer.

## What is preserved from the old path

- SEMSE as ecosystem, not just isolated app;
- vision first;
- marketplace + operations + trust direction;
- evidence, escrow and dispute as core trust primitives;
- multi-surface growth potential;
- anti-chaos discipline around source of truth.

## What changes with the new path

- user modeling becomes group-first;
- permissions become module/action based;
- toolkits become explicit;
- navigation becomes role-aware;
- backend RBAC and frontend shell can align to the same structure.

## Design rule

Do not rebuild from zero.
Map, absorb, refine, then migrate.

## Migration rule

Every major existing piece must be classified as one of these:
- preserve as canonical;
- preserve as reference;
- absorb into new architecture;
- phase out later.
